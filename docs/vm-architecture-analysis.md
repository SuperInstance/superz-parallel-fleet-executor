## Introduction

Virtual machine architecture is a foundational design decision that shapes every aspect of a runtime's performance, code density, and implementation complexity. The two dominant paradigms — stack-based and register-based — represent fundamentally different approaches to how intermediate bytecode instructions operate. Stack-based VMs, exemplified by the JVM and the CPython interpreter, use an implicit operand stack where instructions consume and produce values by pushing to and popping from a LIFO data structure. Register-based VMs, such as Dalvik (Android's original runtime) and Lua 5.0+, instead use explicit virtual registers encoded directly within each instruction, eliminating the need for frequent stack manipulation. The choice between these models is not purely academic; it directly impacts instruction dispatch speed, bytecode size, compiler complexity, and the difficulty of targeting the VM from source languages. This analysis examines the core tradeoffs to inform architecture decisions when designing or evaluating a bytecode runtime.

## Execution Model Comparison

In a stack-based VM, every arithmetic or logical operation implicitly consumes its operands from the top of the stack and pushes its result back. For example, computing `a + b` requires three instructions: `LOAD a`, `LOAD b`, and `ADD`. The stack pointer acts as a hidden register that the VM manages automatically, which simplifies instruction encoding — most instructions need only an opcode byte with zero or one operand. This simplicity makes stack-based bytecodes compact and the interpreter loop straightforward to implement, since operand location is always implicit. However, the cost manifests as excessive memory traffic: every operand movement (push, pop, dup, swap) generates a memory read or write to the stack, and on real hardware these accesses compete with the data cache for space and bandwidth.

Register-based VMs encode operand locations directly within each instruction. The same `a + b` computation becomes a single `ADD r1, r2, r3` instruction, where `r1` and `r2` are source registers and `r3` is the destination. This eliminates intermediate stack manipulation entirely, reducing the total instruction count by roughly 20-40% for typical programs. The tradeoff is that each instruction now carries more bits of operand information, which increases per-instruction size and demands a more complex instruction decoder. Additionally, the compiler or bytecode generator must perform register allocation — assigning source-language temporaries to a finite set of virtual registers — which adds front-end complexity that stack-based designs avoid entirely.

## Performance and Code Density Tradeoffs

Performance differences between the two architectures are subtle and context-dependent. Register-based VMs tend to execute fewer total instructions, which reduces dispatch overhead in the interpreter loop. Since each instruction dispatch involves decoding the opcode and branching to a handler, fewer instructions means fewer branch mispredictions and less indirect jump overhead. Empirical studies, including the well-known analysis by Shi et al. (2005) that motivated Dalvik's register-based design, found that register VMs can reduce instruction count by approximately 25% while increasing total bytecode size by only about 10%. The net effect is often a performance improvement in the interpreter, since the cost of dispatching fewer instructions outweighs the cost of wider instruction decoding.

However, code density — total bytecode size — favors stack-based designs for constrained environments. JVM bytecode is notably compact because most instructions are a single byte; this made it practical for network distribution in Java's original applet model and remains advantageous for embedded systems with limited storage. Register-based bytecodes require additional bytes per instruction to encode register indices, which inflates binary size. JIT compilation complicates this picture further: once a JIT compiler transforms bytecode into native machine code, the original VM architecture becomes largely irrelevant to execution performance, since native code is inherently register-based regardless of the source IR. This means the architectural choice matters most for interpreters and for the complexity of the JIT frontend.

## Implementation Complexity and Portability

Stack-based VMs are significantly easier to implement and target from multiple source languages. The lack of register allocation means that any compiler front-end can emit stack bytecode without needing to manage a finite register file — the stack is effectively unbounded. This simplicity is why the JVM became a successful multi-language target (Java, Kotlin, Scala, Clojure, etc.) and why WebAssembly initially chose a stack-based design for its MVP specification. Writing an interpreter for a stack VM is also straightforward: the core loop is a simple opcode switch with push/pop operations, and formal verification of the instruction semantics is easier when operand flow is strictly linear.

Register-based VMs demand more from both the toolchain and the runtime. The bytecode emitter must perform register allocation, which is a non-trivial compiler optimization problem. The interpreter must decode variable-length instructions with register fields, and the dispatch table grows larger because more instruction formats exist (e.g., one-register, two-register, and three-register variants of common operations). These costs are real but manageable: Lua's register VM demonstrates that a well-designed register architecture can be implemented in under 500 lines of C. The key consideration is whether the performance gains from reduced instruction count justify the added complexity in the compiler and interpreter, which depends on the deployment target and the maturity of the toolchain ecosystem surrounding the VM.
